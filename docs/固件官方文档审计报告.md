## 🔴 阻断性问题

### 1. IDF 版本依赖未在竞赛配置中强制声明

**文件**：`sdkconfig.defaults.competition`
**问题**：`sdkconfig.defaults` 明确写了 `ESP-IDF v5.5+ required (v5.4 has C5 CSI cache bug on 5GHz, fixed in v5.5)`，但 `sdkconfig.defaults.competition` 没有此注释。

**背景**：这是 ESP-IDF 官方确认的 bug（GitHub Issue #18493）— IDF v5.4 下 C5 的 5GHz CSI IQ buffer 会返回静态数据（`wifi_csi_info_t.buf` 内容不变），v5.5 修复。

**影响**：如果按竞赛配置构建时用了 v5.4，5GHz 频段的 CSI 数据全是垃圾。**即使只用 2.4GHz，v5.4 下也存在不确定性。**

**修复**：
```
# 在 sdkconfig.defaults.competition 头部添加：
# ⚠️ ESP-IDF v5.5+ REQUIRED (v5.4 has C5 CSI cache bug, fixed in v5.5)
# idf.py --version 确认版本后再构建
```

### 2. WiFi 频段模式锁死在 2.4GHz — 5GHz/6GHz 代码永远不会执行

**文件**：`main/main.c` — `wifi_init_sta()`
**代码**：
```c
ESP_ERROR_CHECK(esp_wifi_set_band_mode(WIFI_BAND_MODE_2G_ONLY));
```

**问题**：C5 的 5GHz 和 6GHz CSI 能力完全被禁用了。
- 通道跳跃表 `{1, 6, 11, 36, 40, 44}` 中的 ch36/40/44 永远不会生效
- `csi_collector.c` 中的 6GHz 频率计算代码 (`5950 + channel * 5`) 死代码
- `acquire_csi_su/mu` HE 帧采集只能抓到 2.4GHz 的 HE 帧

**官方参考**：esp-csi 示例中 C5 target 使用 `WIFI_BAND_MODE_AUTO` 或 `WIFI_BAND_MODE_5G_ONLY`。

**修复**：
```c
// 修改为：
ESP_ERROR_CHECK(esp_wifi_set_band_mode(WIFI_BAND_MODE_AUTO));
// 或竞赛场景建议 (5GHz 干扰少)：
// ESP_ERROR_CHECK(esp_wifi_set_band_mode(WIFI_BAND_MODE_5G_ONLY));
```

**注意**：如果改为 5GHz/双频，必须确认 IDF v5.5+（见问题 #1）。

## 🟡 应该修复

### 3. Kconfig WiFi 信道范围限制为 2.4GHz

**文件**：`main/Kconfig.projbuild`
**代码**：
```
range 1 13
```
**问题**：C5 支持 5GHz (ch 36–177) 和 6GHz (ch 1–233)，但 Kconfig 范围只有 1–13。要用 5GHz 必须通过 NVS 运行时覆盖。

**修复**：
```
# C5 target 下扩展范围
#if CONFIG_IDF_TARGET_ESP32C5
    range 1 233
#else
    range 1 13
#endif
```

### 4. 竞赛 SSID 使用旧项目名

**文件**：`sdkconfig.defaults.competition`
```
CONFIG_CSI_WIFI_SSID="RuView-Triage"
```
**修复**：改为 `CONFIG_CSI_WIFI_SSID="WCES"` 与项目命名一致。

### 5. CSI 回调中栈上分配大缓冲区 — 需确认栈大小

**文件**：`main/csi_collector.c` — `wifi_csi_callback()`
**代码**：
```c
uint8_t frame_buf[CSI_MAX_FRAME_SIZE];  // 4116 字节在栈上
```
**分析**：CSI 回调运行在 WiFi 任务上下文中。C5 的 WiFi 任务栈由 ESP-IDF 管理（默认约 3KB），但 `frame_buf` 是 4116 字节……实际上 `esp_wifi_set_csi_rx_cb` 的回调是在 WiFi 驱动内部线程中调用的，不是 main task。需要确认：
- `CONFIG_ESP_WIFI_TASK_STACK_SIZE` 是否足够（默认 5120，加上其他开销可能边缘）
- 或者将 `frame_buf` 改为 `static` 分配

**建议**：改 `static` 分配 -> `static uint8_t frame_buf[CSI_MAX_FRAME_SIZE]` 避免栈溢出风险。

### 6. 缺少 ESP-IDF 版本编译期检查

**文件**：`sdkconfig.defaults.competition` / `CMakeLists.txt`
**问题**：没有编译期强制检查 IDF 版本 >= v5.5。

**修复**：在 `CMakeLists.txt` 中添加：
```cmake
if(CMAKE_C_COMPILER_VERSION VERSION_LESS 5.5)
    message(FATAL_ERROR "ESP-IDF v5.5+ required for ESP32-C5 CSI (v5.4 has 5GHz CSI cache bug)")
endif()
```

### 9. 子载波数计算 — 正确 ✅

`n_subcarriers = (info->len - iq_offset) / 2`（单天线，每 I/Q 对 2 字节）。C5 最大 242 子载波 (HE20)，C6 最大 484 (HE40)。缓冲区预留 4116 字节。

### 10. 频率计算公式 — 正确 ✅

| 频段 | 公式 | 验证 |
|------|------|:--:|
| 2.4GHz (ch1-13) | `2412 + (ch-1)*5` | ✅ |
| 2.4GHz (ch14) | `2484` | ✅ |
| 5GHz (ch36-177) | `5000 + ch*5` | ✅ |
| 6GHz (ch1-233) | `5950 + ch*5` | ✅ |

### 11. ADR-018 帧格式 — 正确 ✅

20字节头 + I/Q 数据，与 Rust 服务端 `parse_esp32_frame()` 完全对齐（已在 PROGRESS.md 中审计过字节偏移修复）。

### 12. 边缘计算常量 — 正确 ✅

| 常量 | C5 值 | 理由 |

| `EDGE_MAX_SUBCARRIERS` | 512 | 安全裕度 > 484 |

### 13. Promiscuous 模式 — 正确 ✅

必须启用才能接收所有帧的 CSI，与 esp-csi 官方示例一致。
