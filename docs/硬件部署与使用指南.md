### 1.3 数据流

```
ESP32-C5 ×3                    RZ/G2L (sensing-server)                浏览器
─────────────    ──────────────────────────────────────    ──────────────────
CSI 采集          UDP :5005 →
                  parser::parse_esp32_frame()
                    → Esp32Frame { amplitudes, phases, rssi, node_id... }

                  signal_processing::extract_features_from_frame()
                    → 运动检测、存在检测、特征提取

                  state_ops::smooth_and_classify()
                    → 自适应基线 + 消抖分级

                  VitalSignDetector::process_frame()
                    → VitalSigns { breathing_rate_bpm, heart_rate_bpm,
                                   confidence, signal_quality }

                  TriageEngine::process()
                    → TriageUpdate { survivors, assessment, alerts }

                  EdgeModuleEngine::process_frame()
                    → Vec<EdgeAlert> (19 个边缘模块并行)

                  signal_processing::generate_signal_field()
                    → 20×20 信号场热力图

                  构造 SensingUpdate {
                    vital_signs,
                    triage_update,
                    wasm_alerts,
                    features,
                    classification,
                    signal_field
                  }
                                                          WebSocket :8765 /ws/sensing →
                                                          triage.html 渲染
                                                            ├─ 伤员地图
                                                            ├─ 统计卡片
                                                            ├─ 伤员卡片
                                                            └─ 告警列表
```

### 第 2 步：编译并烧录 ESP32-C5 固件（3 个节点）

#### 2.1 ESP-IDF 环境

**已安装环境 (Windows):**
- ESP-IDF: `C:\esp\v6.0.1\esp-idf`
- RISC-V 工具链: `C:\Espressif\tools\riscv32-esp-elf\esp-15.2.0_20251204`
- 激活: 开始菜单 → `ESP-IDF 6.0.1 PowerShell` 或手动 `C:\esp\v6.0.1\esp-idf\export.ps1`

> ⚠️ ESP-IDF v6.0 对 ESP32-C5 提供完整稳定支持（v5.4 有 C5 5GHz CSI 缓存一致性 bug）。v6.0 有多个 breaking changes（`ESP_IF_WIFI_STA`→`WIFI_IF_STA`、`WIFI_BW_HT40`→`WIFI_BW40`、mbedtls v4 等），固件源码已完成适配。

#### 2.2 修改统一配置文件

编辑项目根目录的 [wces.config.toml](../wces.config.toml)，重点修改以下部分：

```toml
# ============================================================================
# 固件 — ESP32-C5 节点配置
# ============================================================================
[firmware]
node_id = 1                     # ← 节点 1→1, 节点 2→2, 节点 3→3
wifi_ssid = "WCES"              # ← 你的 WiFi SSID
wifi_password = ""              # ← WiFi 密码（开放网络留空）
target_ip = "192.168.1.100"     # ← RZ/G2L 的 IP（非路由器！）

[firmware.edge]
tier = 2                        # 0=原始透传, 1=基础DSP, 2=完整管线（推荐）

[firmware.hop]
enabled = true                  # 启用多频段信道跳跃
channels = [1, 6, 11, 36, 40, 44]  # 2.4G + 5G

# ============================================================================
# 部署 — RZ/G2L 目标
# ============================================================================
[deploy]
rz_ip = "192.168.1.100"         # RZ/G2L 的 IP
rz_user = "root"

# ============================================================================
# 竞赛 — 网络拓扑
# ============================================================================
[competition]
node1_ip = "192.168.1.10"
node2_ip = "192.168.1.11"
node3_ip = "192.168.1.12"

# ============================================================================
# 烧录 — 串口配置
# ============================================================================
[flash]
ports = ["COM3", "COM4", "COM5"]  # 三个节点的串口号
baud = 921600
```

#### 2.3 为每个节点编译并烧录

**当前工作流（一键构建+烧录）：**

```powershell
# 激活 ESP-IDF 环境（开始菜单 → ESP-IDF 6.0.1 PowerShell）

cd D:\CODING\Repository\WCES

# ── 节点 1 ──
.\apply-config.ps1 -NodeId 1            # 生成 sdkconfig.defaults
cd firmware\esp32-c5-csi-node
.\build_firmware_c5.ps1 -FlashPort COMx -Clean   # 全量编译+烧录

# ── 节点 2 (首次需 erase-flash，后续只需 flash) ──
cd ..\..
.\apply-config.ps1 -NodeId 2
cd firmware\esp32-c5-csi-node
idf.py -p COMx erase-flash               # 首次擦除
idf.py -p COMx flash                     # 烧录

# ── 节点 3 ──
cd ..\..
.\apply-config.ps1 -NodeId 3
cd firmware\esp32-c5-csi-node
idf.py -p COMx erase-flash
idf.py -p COMx flash
```

> ⚠️ **COM 口**: 实际串口号通过设备管理器确认。三块全新空片首次烧录前必须 `erase-flash`。
> 
> ⚠️ **板子卡下载模式**: 如果烧录失败且串口显示 `waiting for download`，按板子上的 RST (EN) 按钮复位。如果板子在重启循环中，按住 BOOT → 按 RST → 松开 BOOT 进入下载模式，然后 `erase-flash` + `flash`。

#### 2.4 验证固件

烧录后通过串口监控查看启动日志：

```powershell
idf.py -p COM3 monitor
```

正常启动应看到：
```
I (xxx) wifi: Connected to WCES
I (xxx) csi: CSI callback registered
I (xxx) udp: Streaming to 192.168.1.100:5005
```

### 第 3 步：设置 RZ/G2L

#### 3.1 烧录 Yocto Linux 系统镜像

1. 从[瑞萨官网](https://www.renesas.com)下载 **RZ/G2L Verified Linux Package v3.0.7**：
   - Yocto Recipe Package: `RTK0EF0045Z0021AZJ-v3.0.7.zip`
   - 图形库（可选）: `RTK0EF0045Z14001ZJ-<version>.EN.zip`
   - 视频编解码库（可选）: `RTK0EF0045Z16001ZJ-<version>.EN.zip`

2. 参考 [RZ/G2L Startup Manual](https://www.manualslib.com/manual/3753954/Renesas-Rz-G2l.html) 构建镜像：

   ```bash
   # 构建环境要求：Ubuntu 22.04，至少 100GB 磁盘
   sudo apt-get install gawk wget git-core diffstat unzip texinfo gcc-multilib \
     build-essential chrpath socat cpio python3 python3-pip python3-pexpect \
     xz-utils debianutils iputils-ping libsdl1.2-dev xterm p7zip-full \
     libyaml-dev libssl-dev bmap-tools

   # 解压 Yocto Recipe Package
   mkdir ~/rzg_vlp_v3.0.7 && cd ~/rzg_vlp_v3.0.7
   unzip ./RTK0EF0045Z0021AZJ-v3.0.7.zip
   tar zxvf ./RTK0EF0045Z0021AZJ-v3.0.7/rzg_vlp_v3.0.7.tar.gz

   # 构建 Weston 镜像
   TEMPLATECONF=$PWD/meta-renesas/meta-rzg2l/docs/template/conf/ \
     source poky/oe-init-build-env build
   MACHINE=smarc-rzg2l bitbake core-image-weston
   ```

3. 将生成的镜像烧录到 microSD 卡

4. 将 microSD 插入 RZ/G2L，USB-C 供电，启动

#### 3.2 获取交叉编译工具链

**方式 A（推荐—简单直接）：使用通用 ARM64 交叉编译器**

```bash
# 在开发电脑上（Ubuntu/WSL）
sudo apt install gcc-aarch64-linux-gnu g++-aarch64-linux-gnu
rustup target add aarch64-unknown-linux-gnu

# 配置 cargo linker（在项目 .cargo/config.toml 中）
mkdir -p .cargo
cat >> .cargo/config.tom << 'EOF'
[target.aarch64-unknown-linux-gnu]
linker = "aarch64-linux-gnu-gcc"
EOF
```

**方式 B：使用 Yocto SDK（完整 sysroot）**

```bash
# 在构建环境中生成 SDK
cd ~/rzg_vlp_v3.0.7/build
MACHINE=smarc-rzg2l bitbake core-image-weston -c populate_sdk

# 安装 SDK
sudo sh ./tmp/deploy/sdk/poky-glibc-x86_64-core-image-weston-aarch64-smarc-rzg2l-toolchain-3.1.XX.sh
# 默认安装到 /opt/poky/3.1.XX/

# 激活 SDK 环境
source /opt/poky/3.1.XX/environment-setup-aarch64-poky-linux
```

> 方式 A 更简单，但某些 C 依赖（如 OpenSSL）可能需要额外交叉编译。方式 B 提供完整的 sysroot，但需要先完成 Yocto 构建。

#### 3.3 配置 RZ/G2L 网络

```bash
# SSH 登录 RZ/G2L（默认 root 无密码）
ssh root@192.168.1.100

# 设置静态 IP
cat > /etc/systemd/network/10-eth0.network << 'EOF'
[Match]
Name=eth0

[Network]
Address=192.168.1.100/24
Gateway=192.168.1.1
DNS=192.168.1.1
EOF

systemctl restart systemd-networkd

# 确认系统信息
uname -a       # 应显示 aarch64 Linux
free -h        # 确认 ~1GB RAM
ip addr        # 确认 IP 为 192.168.1.100
```

## 四、配置详解

所有配置集中在项目根目录的 [wces.config.toml](../wces.config.toml) 一个文件中：

### 4.1 固件配置 `[firmware]`

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `node_id` | `1` | 节点编号 (1/2/3)，三个节点唯一区别 |
| `wifi_ssid` | `"WCES"` | WiFi SSID |
| `wifi_password` | `""` | WiFi 密码（空 = 开放网络） |
| `csi_channel` | `6` | 主信道 (2.4G: 1-13, 5G: 36-177) |
| `csi_band` | `"AUTO"` | 频段：AUTO / 2G_ONLY / 5G_ONLY |
| `target_ip` | `"192.168.1.100"` | RZ/G2L 的 IP |
| `target_port` | `5005` | UDP 发送端口 |

### 4.2 边缘处理配置 `[firmware.edge]`

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `tier` | `2` | 0=原始透传, 1=基础DSP, 2=完整管线 |
| `vital_interval_ms` | `1000` | 生命体征包发送间隔 |
| `top_k_count` | `8` | Top-K 高方差子载波数 |
| `presence_thresh` | `0.0` | 存在检测阈值 (0=自动校准) |
| `fall_thresh` | `400.0` | 跌倒检测阈值 (rad/s²) |
| `vital_window` | `64` | BPM 相位历史窗口帧数 |

### 4.3 信道跳跃配置 `[firmware.hop]`

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `enabled` | `true` | 启用多频段信道跳跃 |
| `channels` | `[1, 6, 11, 36, 40, 44]` | 跳跃信道列表 (2.4G + 5G) |
| `dwell_ms` | `50` | 每信道停留时间 |

### 4.4 服务端配置 `[server]`

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `source` | `"auto"` | `auto`=自动检测 / `esp32`=仅硬件 / `simulate`=纯模拟 |
| `http_port` | `8080` | HTTP 服务端口 |
| `ws_port` | `8765` | WebSocket 端口 |
| `udp_port` | `5005` | UDP CSI 接收端口 |
| `bind_addr` | `"0.0.0.0"` | 绑定地址 |
| `ui_path` | `"../docs/triage-ui"` | UI 文件路径 |
| `log_level` | `"info"` | 日志级别 |

### 4.5 Medical Agent 配置 `[server.agent]`

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `enabled` | `true` | 启用 Medical Agent |
| `mode` | `"agent"` | `agent`=云端LLM+本地降级 / `template-only`=仅模板 |
| `periodic_analysis_secs` | `30` | 周期巡检间隔 (0=仅事件触发) |
| `per_patient_cooldown_secs` | `300` | 同一伤员冷却时间 |
| `fallback_to_template` | `true` | LLM 失败时自动回退模板分析 |

### 4.6 LLM 网关配置 `[server.agent.gateway]`

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `endpoint` | `""` | 空=使用 OpenAI / 可替换为 Ollama/vLLM 端点 |
| `model` | `"gpt-4o-mini"` | 模型名 |
| `api_key` | `""` | 空=从环境变量 `LLM_API_KEY` 读取 |
| `timeout_secs` | `20` | 单次请求超时 |
| `temperature` | `0.3` | 采样温度 (0-2) |

### 4.7 边缘模块开关 `[server.edge_modules]`

19 个边缘检测模块，可按需禁用：

```toml
vital_trend = true               # 生命体征趋势

coherence = true                 # 信号相干性
```

## 五、关键注意事项

### 5.1 ESP-IDF 版本与芯片特性

- **ESP-IDF v6.0.1 已安装并验证**。C5 在 v6.0 中从预览毕业为完整稳定支持
- 工具链：C5 使用 `riscv32-esp-elf` (RISC-V)，不是 xtensa-esp-elf
- **C5 是单核芯片** (1× RISC-V @ 240MHz)，无 PSRAM，400KB SRAM。固件已适配：
  - 所有 `xTaskCreatePinnedToCore` 使用 `tskNO_AFFINITY`
  - WASM3 运行时已禁用（无 PSRAM，160KB×4 arena 不可用）
  - WiFi 缓冲已优化（TX 8 + RX 16，节省 ~44KB 堆）
  - UDP 发送改为异步队列架构，避免阻塞 WiFi 任务

### 5.2 ESP32-C5 CSI API 差异

C5/C6 使用**新的 `wifi_csi_acquire_config_t`**（`.acquire_csi_*` 字段），与 S3/C3 的旧 API（`.lltf_en` 等）不兼容：

| 芯片 | 结构体 | 主要字段 |
|------|--------|----------|
| S3/C3/ESP32 | `wifi_csi_config_t` | `.lltf_en`, `.htltf_en`, `.stbc_htltf2_en`, `.manu_scale` |
| C5/C6/C61 | `wifi_csi_acquire_config_t` | `.acquire_csi_legacy`, `.acquire_csi_ht20`, `.acquire_csi_ht40`, `.acquire_csi_su`, `.acquire_csi_mu` |

项目的 [csi_collector.c](../firmware/esp32-c5-csi-node/main/csi_collector.c) 已通过 `#if CONFIG_IDF_TARGET_ESP32C5` 条件编译自动处理。

### 5.3 WiFi 6 CSI 子载波数

| 模式 | ESP32-S3 | ESP32-C5 | 提升 |
|------|----------|----------|------|
| HT20 | 56 | 242 (HE) | **4.3×** |
| HT40 | 114 | 484 (HE) | **4.2×** |

- 最大帧缓冲 4116 字节（适配 WiFi 6 484 子载波）
- 主任务栈 7168 字节（比 S3 的 8192 略小）
- 帧长 (20MHz): 20 字头 + 子载波 × 4 字节实部/虚部 × 链路数

### 5.4 RZ/G2L 性能限制

| 规格 | 参数 | 影响 |
|------|------|------|
| CPU | Cortex-A55 ×2 @ 1.2GHz | 信号处理 <50ms/帧，实时性足够 |
| RAM | 1GB DDR4 | 不建议本地运行 LLM |
| AI 加速器 | 无 | 全 CPU 推理 |
| GPU | Mali-G31 | 可用于 UI 加速 |

**LLM 建议**：保持 `server.agent.mode = "agent"`（云端 LLM + 本地模板降级），不要在 RZ/G2L 上本地运行 LLM。云端 LLM 推荐使用 `gpt-4o-mini` 或 `qwen2.5-7b`。

### 5.5 网络规划

- 路由器 DHCP 分配 ESP32-C5 节点的 IP
- RZ/G2L 使用静态 IP（`192.168.1.100`），通过网线连接路由器
- 确保路由器的 IP（`192.168.1.1`）不与 RZ/G2L 冲突
- ESP32-C5 节点通过 WiFi 6 连接路由器

### 5.6 故障预案

系统内置的容错机制：
- **2/3 节点正常工作**：丢失 1 个节点不影响整体功能（三角定位精度略降）
- **WiFi 断连自动重连**：ESP32-C5 固件支持 10 次重试
- **Medical Agent 降级**：网络中断 → 熔断器 → 本地模板分析
- **超时帧处理**：连续丢失 ≥30 帧 → 标记 Unknown → 超时清理

## 七、快速操作速查

```bash
# ── 模拟模式（无硬件） ──
cd rust-server
cargo run -p wifi-densepose-sensing-server -- --source simulate --ui-path ../docs/triage-ui --bind-addr 0.0.0.0 --http-port 8080

# ── 编译 ESP32-C5 固件 ──
cd firmware/esp32-c5-csi-node
idf.py set-target esp32c5
idf.py build

# ── 烧录节点 1 ──
idf.py -p COM3 flash

# ── 交叉编译 Rust 服务端 ──
cd rust-server
cargo build --target aarch64-unknown-linux-gnu --release

# ── 部署到 RZ/G2L ──
scp target/aarch64-unknown-linux-gnu/release/wifi-densepose-sensing-server root@192.168.1.100:/opt/WCES/sensing-server
scp -r ui/ root@192.168.1.100:/opt/WCES/
scp -r docs/triage-ui/ root@192.168.1.100:/opt/WCES/

# ── 在 RZ/G2L 上启动 ──
ssh root@192.168.1.100
cd /opt/WCES && ./deploy.sh

# ── 仪表盘 ──
# http://192.168.1.100:8080/ui/triage.html
```
