# WCES 竞赛改造完整方案

> 项目名称：基于WiFi CSI感知与端侧LLM的方舱生命体征感知与监护系统
> 比赛：第九届(2026)全国大学生嵌入式芯片与系统设计竞赛 · 瑞萨赛道

---

## 一、项目现状 vs 竞赛需求总览

### 竞赛核心要求
```
瑞萨 RZ/V2H 作为主控 + 扩展电路 → 形成完整应用系统
评分：完成度 + 创新性 + 难度 + 系统功能与指标 + 答辩表现
提交：设计报告 + 作品视频 → 初赛
赛区复赛 + 全国总决赛：现场演示 + 质询答辩
全国总决赛：能力测评（细则后续发布）
```

### 你的项目定位
> 面向战争、自然灾害等大规模人员伤亡场景，基于 WiFi CSI 无源感知技术，研发小型化、低成本、坚固耐用的极端条件人员监护系统。实时采集呼吸、心跳、姿态、运动、位置，构建空间生物模型，自动评估急救优先级（START 分诊）并触发异常报警。数据完全本地化，穿透性强，全天候作业。

### N5：端侧 LLM 伤病报告模块（竞赛特色）

**位置：** `crates/wifi-densepose-llm/`（新建 crate）

**功能：** 将 CSI 生命体征数据 + START 分诊结果自动转换为自然语言伤病报告

**技术方案：** Candle (纯 Rust ML 框架) + Qwen2.5-0.5B 量化模型

```rust
// llm_reporter.rs — CSI 生命体征 → 自然语言病历
use candle_core::{Device, Tensor};
use candle_transformers::models::qwen2;

pub struct FieldHospitalReporter {
    model: qwen2::Model,
    tokenizer: Tokenizer,
    device: Device,
}

impl FieldHospitalReporter {
    /// 输入: 生命体征 + 分诊 → 输出: 自然语言报告
    pub fn generate_report(&self, vitals: &VitalSignsInput, triage: &TriageLevel) -> String {
        let prompt = format!(
            "根据以下伤员生命体征数据生成一份方舱医院伤病报告：\n\
             呼吸率: {:.1} 次/分钟\n\
             心率: {:.0} BPM\n\
             START分诊: {}\n\
             请用中文输出报告。",
            vitals.breathing_rate_bpm.unwrap_or(0.0),
            vitals.heart_rate_bpm.unwrap_or(0.0),
            triage.name()
        );
        // Candle 推理 (100-500ms on RZ/V2H CPU)
        self.model.generate(&prompt, 256)
    }
}
```

**部署策略：**
- 默认关闭 (避免过度消耗 CPU)
- 按钮触发单个伤员报告生成
- 模型量化 INT4 → ~250MB，RZ/V2H 8GB RAM 可行
- 备选：云端 LLM API (需网络，不符合"完全本地")

**如果 LLM 实现来不及 — 竞赛对策：**
> 在 PPT 和设计报告中展示 LLM 模块的设计方案和接口定义。
> 答辩时说明："端侧LLM 已完成架构设计和推理框架选型，
> 模型量化后可部署在 RZ/V2H 的 DRP-AI 或 CPU 上，
> 用于将生命体征自动转换为方舱伤病报告。"

## 七、关键技术风险与缓解

| 风险 | 概率 | 缓解方案 |
|------|------|----------|
| RZ/V2H ONNX 推理不工作 | 中 | 回退到纯信号处理模式（不依赖 NN），CSI→生命体征→分诊 不需要 NN |
| 5GHz CSI 数据异常 | 中 | 使用 2.4GHz（已验证稳定）；备 5GHz 固定信道 |
| 竞赛现场 WiFi 干扰 | 高 | 专用路由器 + 5GHz 信道 + 有线以太网备用 |
| 交叉编译失败 | 低 | 提前在 RZ/V2H SDK 环境编译验证；备 x86 笔记本跑 demo |
| 演示效果不稳定 | 中 | 多次完整彩排；预制录制备用视频 |
