## 一、项目概述

### 1.1 这是什么系统？

**WCES** 是一套面向 **方舱医院/灾难救援场景** 的 **无源 WiFi 生命体征感知与监护系统**。

- **不依赖摄像头**：利用现有 WiFi 信号的 CSI（信道状态信息）感知人体
- **全本地化**：数据不出方舱，无需云服务，适合野外/战区/灾区
- **自动分诊**：实时监测伤员生命体征，自动执行 START 分诊协议

### 1.2 核心价值

| 维度 | 说明 |
|------|------|
| **穿墙感知** | WiFi 信号可穿透墙体，感知废墟下的生命迹象 |
| **隐私保护** | 无摄像头，不采集图像，保护伤员隐私 |
| **无人值守** | 自动监测、自动分诊、自动告警，减轻医疗人员负担 |
| **极端环境** | 全本地部署，不依赖外部网络，适合战区/灾区 |
| **低成本** | 仅需 ESP32-C5 × 3 + RZ/V2H，成本远低于医疗监护仪 |

### 2.2 数据流

```
ESP32-C5 CSI 帧 (ADR-018 二进制格式)
    ↓ UDP :5005
解析帧: 振幅 + 相位 + RSSI + 元数据
    ↓
CSI 信号预处理:
  ├─ Hampel 滤波 (去异常值)
  ├─ 子载波加权融合
  └─ 静态成分抑制 (EMA 高通)
    ↓
生命体征提取:
  ├─ 呼吸率: IIR 带通 (0.1-0.5Hz) + 零交叉 → 6-30 BPM
  └─ 心率: 相位相干性 + IIR 带通 (0.8-2.0Hz) + 自相关 → 40-120 BPM
    ↓
START 分诊:
  ├─ Immediate (红): RR>30 或 RR<10, HR>120 或 HR<40
  ├─ Delayed (黄): 异常但未到临界
  ├─ Minor (绿): 正常 + 可移动
  └─ Deceased (黑): 无生命体征
    ↓
伤员追踪:
  ├─ 匹配/创建伤员记录
  ├─ Kalman 滤波平滑生命体征
  ├─ RSSI → 距离 → 三角定位
  └─ 恶化检测 (分诊等级跨级上升 → 告警)
    ↓
SensingUpdate JSON (WebSocket :8765)
  ├─ vital_signs: {呼吸率, 心率, 置信度}
  ├─ triage_update: {伤员列表, 分诊统计, 告警}
  ├─ pose_keypoints: [可选] 17 个 COCO 关键点
  └─ signal_field: [可选] 3D 高斯点渲染
    ↓
浏览器 UI 渲染:
  ├─ 分诊仪表盘: 2D 地图 + 伤员卡片 + 统计
  └─ 3D 可视化: Three.js 骨架 + 高斯点
```

## 三、核心技术原理

### 3.1 WiFi CSI 感知原理

**CSI (Channel State Information)** 描述 WiFi 信号在发射器和接收器之间的传播特性：

```
发射信号 → 直射路径 + 反射路径 + 散射路径 → 接收信号
                              ↓
            人体存在时 → 额外的反射路径 → CSI 变化
```

**CSI 包含每个子载波的**：
- **振幅 (Amplitude)**：信号强度，受人体遮挡和反射影响
- **相位 (Phase)**：信号传播路径长度，对人体微小位移敏感

**ESP32-C5 优势 (WiFi 6)**：
- 支持 **HE-LTF** (High Efficiency Long Training Field)
- 提供 **484 个子载波** (是 ESP32-S3 114 个的 4.2 倍)
- 2.4GHz + 5GHz 双频
- **更高子载波数 → 更高空间分辨率 → 更精确的人体感知**

### 3.2 呼吸率检测 (`wifi-densepose-vitals/src/breathing.rs`)

```
CSI 振幅 → 多子载波加权融合
         → 2阶 IIR 谐振带通滤波器 (0.1 - 0.5 Hz)
         → 零交叉计数 → 呼吸率 (BPM)
```

**核心代码**：

```rust
// 滤波核心: y[n] = (1-r)*(x[n] - x[n-2]) + 2*r*cos(w0)*y[n-1] - r²*y[n-2]
let output = (1.0 - r) * (input - state.x2) 
           + 2.0 * r * cos_w0 * state.y1 
           - r * r * state.y2;
```

| 参数 | 值 | 说明 |
|------|-----|------|
| 频率范围 | 0.1 - 0.5 Hz | 对应 6-30 次/分钟 |
| 窗口 | 30 秒 | 需要 10 秒数据预热 |
| 子载波数 | 56 (ESP32 默认) | 加权融合 |
| 置信度 | SNR/5 | 映射到 [0, 1] |

**算法优势**：
- IIR 滤波器计算量 O(1)，适合实时嵌入式系统
- 零交叉法比 FFT 快约 5 倍
- 加权子载波融合利用空间多样性提高鲁棒性

### 3.3 心率检测 (`wifi-densepose-vitals/src/heartrate.rs`)

```
CSI 振幅 + 相位 → 子载波相位相干性加权
               → 2阶 IIR 带通滤波器 (0.8 - 2.0 Hz)
               → 自相关峰值检测 → 心率 (BPM)
```

**关键挑战**：心脏跳动引起的体表位移仅 0.1-0.5mm，是呼吸位移 (1-5mm) 的 1/10。

**应对方案 − 相位相干性加权**：
- 相邻子载波相位差小 = 高相干性 = 来自同一体表区域
- 相干性高的子载波加权更高 → 增强微小心脏信号

**自相关峰值检测**：
- 在 0.8-2.0 Hz (48-120 BPM) 范围搜索
- 找自相关函数的最大峰值 → 对应心率周期

| 参数 | 值 | 说明 |
|------|-----|------|
| 频率范围 | 0.8 - 2.0 Hz | 对应 48-120 次/分钟 |
| 窗口 | 15 秒 | 需要 5 秒数据预热 |
| 最小子载波数 | 4 | 不足则降级为 Unreliable |
| 算法 | 自相关 | 比 FFT 对弱信号更鲁棒 |

### 3.4 START 分诊 (`wifi-densepose-sensing-server/src/mat_pipeline.rs`)

**START (Simple Triage and Rapid Treatment) 协议**：

```
                ┌─────────────────────┐
                │   伤员生命体征输入    │
                │ 呼吸率 + 心率 + 运动  │
                └──────────┬──────────┘
                           │
                ┌──────────▼──────────┐
                │   信号质量 >= 0.05?  │
                └──────┬───────┬──────┘
                   Yes │       │ No
                ┌──────▼──┐   ┌▼────────┐
                │ 有生命体征?│   │ Unknown │
                └──┬───┬──┘   │  (灰色)  │
               Yes │   │ No   └─────────┘
           ┌───────▼┐  └─► ┌──────────┐
           │ RR>30  │      │ Deceased │
           │ RR<10? │      │  (黑色)  │
           └──┬───┬─┘      └──────────┘
          Yes │   │ No
       ┌──────▼┐ ┌▼─────────┐
       │ Imm.  │ │ HR>120   │
       │ (红)  │ │ HR<40?   │
       └───────┘ └──┬───┬───┘
                 Yes │   │ No
              ┌──────▼┐ ┌▼───────────┐
              │ Imm.  │ │ 运动 >0.3  │
              │ (红)  │ │ + 体征正常? │
              └───────┘ └──┬───┬─────┘
                       Yes │   │ No
                     ┌─────▼┐ ┌▼─────────┐
                     │Minor │ │RR 12-20  │
                     │(绿)  │ │HR 60-100?│
                     └──────┘ └──┬───┬───┘
                            Yes  │   │ No
                          ┌──────▼┐ ┌▼───────┐
                          │Minor  │ │Delayed │
                          │(绿)   │ │ (黄)   │
                          └───────┘ └────────┘
```

**伤员追踪**：
- 匹配策略：按 node_id + 时间窗口匹配
- 平滑：最近 5 个值的均值
- 位置：RSSI → 距离 → 3 节点三角定位
- 恶化检测：分诊等级跨级上升 ≥ 2 级 → 触发告警
- 超时清理：30 秒无更新 → 标记为 lost

### 5.3 关键技术风险

| 风险 | 概率 | 缓解方案 |
|------|:----:|----------|
| RZ/V2H ONNX 推理不工作 | 中 | 回退到纯信号模式 (不需 NN) |
| 5GHz CSI 数据异常 | 中 | 使用 2.4GHz (已验证), 备 5GHz 固定信道 |
| 竞赛现场 WiFi 干扰 | 高 | 专用路由器 + 5GHz + 有线以太网备用 |
| 交叉编译失败 | 低 | 提前在 SDK 环境验证, 备 x86 笔记本跑 demo |
| LLM 推理速度不足 | 低 | INT4 量化 + 三级降级 (LLM→模板→最小模式) + 流式输出 |
